  1. .claude 폴더 구조와 각 역할

  agents/ — 사용자 정의 Sub Agent

  - 역할: 사용자가 직접 만든 전문 에이전트 정의 파일 (.md)
  - 현재 4개: csharp-coding-expert, nsis-installer-expert, python-expert,
  script-specialist
  - Claude Code가 Agent 도구를 사용할 때, 이 .md 파일의 description을 보고
  자동으로 매칭하여 해당 에이전트를 실행합니다
  - 모델, 색상, 전문 영역 등을 지정할 수 있음 (예: model: sonnet)

  commands/ — 사용자 정의 Slash Commands

  - 역할: /convert, /review, /worklist 같은 커스텀 명령어 정의
  - /sc:* 같은 SuperClaude 명령어도 여기에 포함
  - Skill 도구를 통해 실행됨

  teams/ — Agent Teams 설정

  - 역할: 여러 에이전트가 협업하는 팀 단위 작업 설정 저장
  - gpsecurity-migration 폴더는 SimulationDAQ GpSecurity v1→v2 마이그레이션 작업
   시 자동 생성된 것입니다
  - 구조:
    - config.json: 팀 이름, 리드 에이전트, 멤버 정보
    - inboxes/: 각 에이전트 간 메시지 교환 파일 (team-lead,
  agent-infrastructure, agent-migration 등)
  - 팀 리드가 작업을 분배하고, 각 에이전트의 inbox를 통해 지시/결과를 주고받는
  구조

  plugins/ — 플러그인 관리

  - 역할: 외부 플러그인 설치/관리 (SuperClaude 등)
  - installed_plugins.json에 설치된 플러그인 목록 관리
  - marketplaces/: 플러그인 마켓플레이스 정보

  tasks/ — Task 실행 기록

  - Agent 도구로 생성된 sub-agent 작업 기록 (UUID별 폴더)
  - gpsecurity-migration 폴더도 여기에 있음 — Teams 작업의 실행 데이터

  2. 기존 Sub Agent가 실행되는 문제와 병렬 제약

  예, 제약이 있을 수 있습니다.

  Claude Code가 Agent 도구를 호출할 때:
  1. agents/ 폴더의 .md 파일 description을 확인
  2. 매칭되는 에이전트가 있으면 해당 에이전트의 시스템 프롬프트와 모델로 실행
  3. Teams에서 생성한 에이전트와 이름/역할이 겹치면 의도치 않게 기존 sub agent가
   선택될 수 있음

  병렬 실행 제약:
  - 기존 sub agent가 model: sonnet으로 설정되어 있으면, Teams에서 Opus를
  의도했어도 Sonnet으로 실행됨
  - 기존 agent의 시스템 프롬프트가 Teams 작업 맥락과 맞지 않을 수 있음

  3. 기존 Sub Agent 삭제 여부 — 장단점

  삭제하는 경우 (권장)

  ┌───────────────────────────────────┬─────────────────────────────────────┐
  │               장점                │                단점                 │
  ├───────────────────────────────────┼─────────────────────────────────────┤
  │ Teams가 자유롭게 최적의           │ 특정 분야 전문성 프롬프트를 매번    │
  │ 에이전트를 자동 구성              │ 재지정해야 함                       │
  ├───────────────────────────────────┼─────────────────────────────────────┤
  │ 이름/역할 충돌 없음               │ -                                   │
  ├───────────────────────────────────┼─────────────────────────────────────┤
  │ 모델 선택이 Teams 설정에 따라     │ -                                   │
  │ 일관적                            │                                     │
  └───────────────────────────────────┴─────────────────────────────────────┘

  유지하는 경우

  ┌──────────────────────────────────────────────┬──────────────────────────┐
  │                     장점                     │           단점           │
  ├──────────────────────────────────────────────┼──────────────────────────┤
  │ 반복적인 단독 작업 시 전문 에이전트 즉시     │ Teams와 충돌 가능성      │
  │ 활용                                         │                          │
  ├──────────────────────────────────────────────┼──────────────────────────┤
  │ 세밀한 시스템 프롬프트 커스터마이징 유지     │ 의도치 않은 에이전트     │
  │                                              │ 매칭                     │
  └──────────────────────────────────────────────┴──────────────────────────┘

  권장 방안

  삭제보다는 이름을 명확히 구분하는 것을 추천합니다:
  - agents/ 폴더의 파일은 단독 작업용 sub agent로만 사용
  - Teams 작업 시에는 Claude가 자동으로 적절한 에이전트를 생성하므로, agents/의
  description에서 Teams 맥락과 겹치지 않도록 scope를 좁히면 됩니다
  - 또는, 잘 사용하지 않는 agent .md 파일은 백업 후 삭제해도 무방합니다

  4. 폴더 간 상호 영향 요약

  commands/  ─── Skill 도구로 실행 (사용자 직접 호출)
      │
  agents/   ─── Agent 도구로 자동 매칭 (description 기반)
      │          ↕ 충돌 가능
  teams/    ─── Agent Teams가 자동 생성/관리 (팀 협업)
      │          └── tasks/ 에 실행 데이터 저장
  plugins/  ─── commands/에 명령어 추가, 독립적 동작

  - commands와 agents는 독립적 (commands는 사용자 호출, agents는 자동 매칭)
  - agents와 teams가 가장 충돌 가능성이 높음 — 같은 Agent 도구를 공유하기 때문
  - plugins는 commands에 새 명령어를 추가하는 방식으로 영향 (SuperClaude의 /sc:*
   명령어 등)
